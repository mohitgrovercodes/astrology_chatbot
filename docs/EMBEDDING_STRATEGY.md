# Embedding Strategy & Retrieval Fitness

> **What we embed, how it gets searched, and whether the choice supports the project's tier-based answer logic.**
> Written as an analysis after the NER catalog work in May 2026. Pairs with `NER_CATALOGS.md` (what goes into the metadata that the retriever filters on) and `INGESTION.md` (how the embeddings get there).

---

## TL;DR

The current embedding setup will serve the chatbot well. Three reasons:

1. **The embedded text is more than just the raw paragraph.** `text_for_embedding` prepends an English `Section:` and an LLM-generated `Summary:` before the body. That summary carries the semantic load even when the body is pure Sanskrit, which is what lets a Devanagari-only book like *Varahmihira Horasastram Vol 1* still be retrievable in response to an English query.
2. **The vectoriser is multilingual.** `gemini-embedding-001` (1536d, cosine) places Sanskrit, IAST, and English in the same space, so `"shani sade sati"`, `"Saturn's seven-and-a-half year transit"`, and `"शनेः साढ़े साती"` all land near each other without a translation step.
3. **Metadata filters do the heavy lifting on entity precision.** After NER canonicalisation, a hard filter like `planets="Saturn"` matches 4,300+ chunks deterministically and the embeddings rank within that subset. This is faster *and* more accurate than asking cosine to find Saturn-related chunks unaided.

Where the strategy is weakest: BM25 keyword recall on Sanskrit queries (the tokenizer is `.lower().split()` which mangles Devanagari) and HyDE latency (one extra LLM call per query). Neither blocks ingestion; both have an obvious upgrade path.

## 1. What goes into a chunk's embedding

Construction lives in `src/rag/preprocessing/chunk_enricher.py:construct_embedding_text`:

```
Section: <chapter / section heading>

Summary: <LLM-generated English summary of the unit>

<combined_text>            # raw display text — English, Sanskrit, or mixed
```

Three knobs shape what the vector ultimately captures:

- **Section prefix.** Acts as a coarse topic prior. Chunks from "Chapter 12 — Marriage" cluster more tightly with marriage queries than they would on body text alone, because every chunk's first line is the same canonical heading.
- **Summary prefix.** An LLM-generated English paraphrase produced during enrichment. This is the single most important design choice in the pipeline. For pure-Devanagari source material (Varahmihira Vol 1: ~99% Sanskrit), the summary is the *only* English signal the embedder sees — and it's prepended to the vector text, so it dominates the early-position attention weights of the embedding model.
- **Body.** Whatever the source produced. For BV Raman or BPHS (Sharma's English-translated edition) this is English plus inline Sanskrit verses. For Varahmihira / Jataka Parijata it is Devanagari.

The shape means: even if a chunk is text-shaped like a Sanskrit shloka, it embeds like an English sentence. That has a few downstream consequences.

### What this gives us
- Cross-lingual retrieval *just works* for English queries against Sanskrit chunks.
- The same chunk is approximately as retrievable in English as in Sanskrit, because both signals are present.
- A user asking "When is my Sade Sati?" can hit a Sanskrit-only Varahmihira passage on Saturn transits without ever typing a Sanskrit word.

### What it costs us
- The summary is generated by an LLM at enrichment time, so summary quality is bounded by that pass. A poor or generic summary handicaps every future retrieval of that chunk.
- BM25 over a Sanskrit query will fail — the index is built from the same `text_for_embedding` string and tokenized with a naïve `.lower().split()`. So `"शनेः साढ़े साती"` will hit semantic recall but not keyword recall.

## 2. How the retriever uses the embeddings

`src/rag/retriever.py:AstrologyRetriever` (and the production-shaped `src/ai/hybrid_retriever.py`) compose four primitives:

| Path | What it does | Strength |
|---|---|---|
| **Semantic** | Embed the query with the same model and run `collection.query(query_embeddings=...)` (cosine). | Generalisation, paraphrase tolerance, cross-lingual recall. |
| **BM25** | In-memory BM25Okapi over the same documents. Tokenised by `.lower().split()`. | Exact technical terms (`navamsha`, `vargottama`, `D-9`). |
| **RRF hybrid** | Reciprocal Rank Fusion of semantic + BM25 with a configurable `semantic_weight` (default 0.7). | Robust to whichever path the query plays into. |
| **HyDE** | Ask the LLM to write a hypothetical answer to the query, embed *that* answer, search with it. | Better recall on under-specified or "advice"-shaped queries that the raw query embedding handles poorly. |

The metadata layer runs *over* these:

```python
def _build_where_clause(filters):
    for key, value in filters.items():
        if key in ("planets", "houses", "signs", "nakshatras"):
            conditions.append({key: {"$contains": value}})
        else:
            conditions.append({key: {"$eq": value}})
```

That `$contains` is what makes the comma-joined NER convention work — `metadata["planets"] = "Sun,Saturn,Moon"` lets a filter `planets="Saturn"` substring-match cleanly. (See `INGESTION.md` for the storage convention. **Don't** ingest via `src/rag/ingest_local.py`; it writes under `entity_planets` with a different separator that this filter never sees.)

## 3. What "tier rules" means here, and whether embeddings help

The codebase uses "tier" in two different senses. Distinguishing them matters for this question.

### 3a. Validation tier rules (`optimized/tiered_rules.json`)

16,526 chart-validation rules grouped into 4 tiers. Selected at runtime by `determine_validation_tier(query_type, question_mode, domain, detail_level)` in `src/orchestration/orchestrator_validation_helpers.py`. Tier 1 is the live-chat budget (cap ~80 rules, severity-filtered); tier 3 is the deep analysis path.

**These rules don't touch embeddings.** They evaluate symbolic patterns against the user's chart (planet-in-house, planet-aspecting-house, lord-of-X-in-Y) via `VedicValidationEngineV2`. The corpus could be empty and the rules would still fire.

So: better embeddings do not directly improve tier-rule scoring. They can indirectly enrich the final answer by providing supporting passages that the LLM weaves into the rule-driven explanation, but the rule firings themselves are independent.

### 3b. Retrieval tier (intent-keyed `top_k`)

The retriever applies an intent-based `top_k` bucket:

```python
intent_top_k_map = {
    'PREDICTION': 15, 'INTERPRETATION': 12,
    'RAG_WITH_CALCULATION': 10, 'RAG_ONLY': 8,
    'LEARNING': 10, 'CHITCHAT': 3,
}
```

This **is** the embeddings' tier. The ranking signal that decides which chunks make it into the bucket is the cosine similarity of the query embedding to each chunk's embedding (plus the BM25/HyDE/metadata overlays). Better embeddings → better ranking → more relevant chunks at each `top_k`.

Two ways the current embedding strategy is well-suited here:
- **Summary-front-loaded embeddings prevent recall collapse on Sanskrit chunks** for English `PREDICTION`/`INTERPRETATION` queries. Without the English summary prefix, a `PREDICTION` query like "When will I get married?" would never surface a Varahmihira passage on 7th-house affairs even though the chunk is highly relevant.
- **NER metadata is the precision rail.** Once `entities.planets` is canonicalised, a query that classifies as `marriage` domain can pre-filter on `planets=Venus,Jupiter,Mars` and `houses=7,2,11` *before* spending cosine budget on irrelevant chunks. The candidate set shrinks from 14k to a few hundred, and the embedding ranks within that.

### 3c. Indirect: better retrieval → better synthesis

The orchestrator's intelligence pipeline (FactorScorer → AnswerPlanner → LLM synthesis) consumes both rule firings *and* retrieved chunks. A retrieval that surfaces a passage on, say, `Viparita Raja Yoga` for a chart that actually has one lets the synthesis layer cite the classical source instead of relying purely on the rule's terse explanation. The yoga NER work directly enables this — before canonicalisation, the chunk could only be found by exact-spelling matches; after canonicalisation, a filter `yogas=Viparita Raja Yoga` lifts every Latin/Devanagari/inflected mention.

## 4. Where the strategy has gaps

| Gap | Symptom | Mitigation |
|---|---|---|
| **BM25 over Sanskrit** | `"शनेः साढ़े साती"` returns zero BM25 hits because tokenisation strips Devanagari without normalising. | Bump `semantic_weight` to 0.9 when the query is non-ASCII (cheap, defensible default). Long-term, a Devanagari-aware tokenizer. |
| **HyDE adds an LLM hop** | One extra Gemini call per query — measurable in p95. | Already on the latency-reduction backlog (#42). Gate HyDE behind an intent / confidence trigger rather than always-on. |
| **No `content_type` / `language` metadata** on current chunks | The intent-conditioned content filter (`PREDICTION` → `interpretation` chunks) is a no-op. Retriever silently skips when the field is absent — defensive, but the feature isn't real yet. | Backfill `content_type` during the next enrichment pass. Add `language="en"/"sa"/"mixed"` based on script ratio. |
| **`expand_context` needs `chunk_index`** | The adjacent-chunk expansion fetches by `(source_book, chapter, chunk_index)`. Most current chunks lack `chunk_index`. | Easy retrofit during the next vector_db_builder run. |
| **Summary dominates the vector** | When the LLM summariser drifts, every retrieval of that chunk drifts with it. No re-summarisation pass is wired. | Spot-check sample summaries during eval. Periodic re-summarisation (cheap, parallelisable). |

None of these block first ingestion. They are the natural follow-up backlog after the corpus is live.

## 5. Bottom line for the user's question

> *"Will the embeddings actually help in giving responses or forming tiered rules?"*

- **Responses, yes — strongly.** The summary-front-loaded, multilingual `gemini-embedding-001` vectors, combined with canonicalised NER metadata for hard filters, are exactly the shape this retriever was designed around. Cross-lingual recall, paraphrase tolerance, and entity-precise filtering all land.
- **Tier rules, no — by design.** The validation tier rules are symbolic and chart-driven; they don't and shouldn't depend on what the corpus says. What the embeddings do is make the *evidence layer alongside* those rule firings sharper, so the answer the user sees pairs each symbolic pattern (rule) with a relevant classical citation (chunk) rather than a generic one.
- **The retrieval tier (intent-keyed top_k), yes.** That tier *is* the embedding rank. Every win in embedding fidelity flows straight into the bucket.
